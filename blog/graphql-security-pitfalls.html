GraphQL Security Pitfalls I Keep Finding in Bug Bounty Programs

GraphQL didn't introduce new vulnerability classes so much as it gave the old ones a new place to hide.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Start with introspection to map the entire schema for free, then check whether authorization is actually enforced per-field and per-resolver instead of just at the top level, and test whether batching or aliasing lets you bypass rate limits.

Teams migrating from REST to GraphQL often bring their REST-era security assumptions with them, and those assumptions don't map cleanly. A REST API with one authorization check per route is a very different shape from a GraphQL API with one resolver per field, each of which needs its own check. That mismatch is where most of what I find actually lives.

1. Introspection: the schema handed to you for free

If introspection is enabled in production — and it still often is — you can request the entire schema in one query, including types, fields, and mutations that were never meant to be public knowledge:

POST /graphql HTTP/1.1
Content-Type: application/json

{"query":"{ __schema { types { name fields { name } } } } "}

This alone regularly reveals admin-only mutations, internal-sounding types, and fields that hint at features not yet exposed in the UI. Even when introspection is disabled, tools like clairvoyance can often reconstruct a usable schema through targeted field-guessing, since GraphQL's error messages are frequently more descriptive than intended.

2. Field-level authorization gaps

This is the big one. Plenty of GraphQL APIs check authorization at the query root — "is this user allowed to query user?" — but not on every nested field within it. That means a query like this can sometimes return data the requester was never supposed to see:

{
  user(id: "4821") {
    name
    email
    billingAddress
    internalRiskScore
    supportTickets { subject, body }
  }
}

If internalRiskScore or supportTickets were meant to be admin-only fields but the resolver for the parent user type is the only place authorization is checked, a regular user asking for those nested fields directly can bypass it entirely. This is functionally an IDOR, just expressed through field selection instead of a URL parameter.

3. Batching and aliasing to bypass rate limits

Rate limiting built for REST usually counts HTTP requests. GraphQL lets a client bundle many logical operations into a single HTTP request using aliases:

{
  attempt1: login(username: "victim", password: "guess1") { token }
  attempt2: login(username: "victim", password: "guess2") { token }
  attempt3: login(username: "victim", password: "guess3") { token }
  ...
}

If the backend only counts this as one request for rate-limiting purposes, that's hundreds of effective attempts hiding inside a request budget meant to allow only a handful. This applies well beyond login — password reset tokens, coupon codes, and OTP verification are all common aliasing targets.

4. Denial of service via nested or recursive queries

GraphQL's ability to nest related objects arbitrarily deep is a feature that doubles as a resource exhaustion vector if the server doesn't cap query depth or computational cost:

{
  user(id: "1") {
    friends {
      friends {
        friends {
          friends { name }
        }
      }
    }
  }
}

Each additional nesting level can multiply the number of database lookups the resolver chain performs. Without query depth limiting, complexity scoring, or pagination enforcement, a handful of deeply nested requests can meaningfully load the backend — worth testing carefully and in a way that respects the program's availability expectations.

5. Excessive data exposure by default

Because a single GraphQL type is often reused across many different queries, it's common for a resolver to fetch and return an entire database row internally, then rely on the client to only ask for the fields it needs. If any part of the response pipeline logs the full resolver output, or if a client library caches the complete response, fields the frontend never displays can still leak.

Tools that help

  • InQL — Burp extension for schema exploration and generating query templates from introspection
  • GraphQL Voyager — visualizes the schema as a graph, useful for spotting types that hint at unreleased features
  • graphql-cop — automated checks for common misconfigurations like introspection exposure and missing query depth limits
Key takeaways
  • Introspection, when enabled, maps the entire attack surface in one request
  • Check authorization at the field level, not just the query root
  • Aliasing can turn one rate-limited request into hundreds of effective attempts
  • Query depth and complexity limits matter as much as authentication does
