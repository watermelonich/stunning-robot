XSS in 2025: Bypassing Filters, CSPs, and Framework Sanitization

React, Vue, and a strict CSP will stop the obvious payloads. They won't stop the ones hiding in your own escape hatches.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Modern frameworks auto-escape by default, which pushes real-world XSS into the places developers explicitly opted out of escaping, or into CSP configurations that look strict but leave one gap. Find the opt-out, find the gap.

"XSS is basically dead now that everything uses React" is a sentence I hear a lot, usually right before I find one. Auto-escaping templates and Content Security Policy raised the floor, but they didn't remove the bug class — they just moved it into more specific, more interesting places.

The three flavors, still alive and well

  • Reflected — input from the request is echoed back into the response unescaped, usually in server-rendered error pages, search results, or legacy endpoints that never got migrated to the new frontend
  • Stored — input is saved and rendered later for other users; profile fields, comments, and support tickets are the classics
  • DOM-based — the vulnerable sink is entirely client-side JavaScript; the payload never has to touch the server to fire

Where frameworks quietly turn escaping off

Every major frontend framework gives developers an explicit way to render raw HTML, because sometimes you genuinely need to. That escape hatch is where I look first:

  • React — dangerouslySetInnerHTML. The name is basically a warning label, and yet it still gets fed user-controlled markdown-to-HTML output more often than you'd expect.
  • Vue — v-html. Same idea, same risk, especially in CMS-style apps rendering "rich text" fields.
  • Angular — bypassSecurityTrustHtml and friends in the DomSanitizer API. If a developer had to explicitly tell Angular to trust something, that's exactly what I want to test.

Search the JS bundle for these strings first. Nine times out of ten it narrows a huge app down to two or three components worth actually testing.

Mutation XSS (mXSS)

Sanitizers work by parsing HTML into a tree, stripping dangerous nodes, and serializing it back to a string. mXSS happens when the browser re-parses that "clean" output differently than the sanitizer expected, and a dangerous tag reassembles itself in the DOM. It's a great example of why testing a sanitizer in isolation isn't the same as testing it inside the actual browser and actual DOM context it runs in — behavior can differ between parsing passes in ways that are easy to miss on paper.

DOM clobbering

If a script does something like document.getElementById('config').value without first checking that the element is what it expects, you can sometimes "clobber" that reference by injecting an <a name="config"> or <form> element earlier in the page, redefining what that global name resolves to. It's a quiet bug class because there's no <script> tag involved at all — just HTML shaping how JavaScript resolves identifiers.

Content Security Policy: read it before you try to break it

A CSP header is a list of promises. My first move on any target is just reading it carefully:

Content-Security-Policy: script-src 'self' 'nonce-r4nd0m123' https://cdn.example.com;
  object-src 'none'; base-uri 'self';

Questions I ask about a policy like this:

  • Is the nonce actually random per-response, or reused/predictable?
  • Does the allow-listed CDN host any JSONP endpoints or old library versions with known sandbox-escaping gadgets?
  • Is base-uri restricted? Without it, an injected <base> tag can hijack relative script paths elsewhere on the page.
  • Is there an unsafe-inline fallback left in for browsers that don't support nonces, that a modern browser will also happily honor if the nonce check fails open?

A policy with strict-dynamic and a properly random nonce is genuinely hard to bypass. A policy with a wildcard subdomain in script-src is a policy with a hole in it — you just have to find what's hosted on one of those subdomains.

A generic, illustrative payload

For a plain reflected sink with no sanitization at all, the textbook proof-of-concept is still the fastest way to confirm the bug exists before investing time in bypassing anything fancier:

<img src=x onerror=alert(document.domain)>

I use alert(document.domain) specifically because it proves the injection executes in the target's origin, which is the detail that actually matters for impact — not just that a dialog box popped up.

Impact, not fireworks An alert() is a proof of concept, not a finding. The report that gets paid explains what an attacker does next: steal a session token, perform actions as the victim via a hidden fetch, or pivot into an internal admin panel. Show the realistic next step, even briefly.

Key takeaways

Key takeaways
  • Grep the bundle for dangerouslySetInnerHTML, v-html, and bypassSecurityTrustHtml first
  • Read the CSP like a contract and look for the clause with the loophole
  • Don't ignore DOM-based sinks just because the network tab looks clean
  • Always demonstrate impact beyond a popup
