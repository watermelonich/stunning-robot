OAuth Misconfigurations: A Bug Bounty Hunter’s Checklist

OAuth is a solid protocol. Most of the bugs I find aren't in the spec — they're in what implementations quietly skip.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Almost every OAuth finding I report comes down to one of three things: loose redirect_uri validation, a missing or unchecked state parameter, or a token that leaks somewhere it shouldn't — a referrer header, a browser history entry, or a log line.

OAuth 2.0 gets a bad reputation for being "insecure," which isn't really fair — the spec, used correctly, is solid. The problem is that it has a lot of optional pieces, and "optional" has a way of turning into "skipped" under a deadline. This is my running checklist for where that shows up.

1. redirect_uri validation

The authorization server should only ever send the authorization code or token to a redirect URI that was pre-registered for that client. When validation is loose, you get an open redirect that leaks live credentials instead of just a boring URL. Variations worth testing:

  • Exact match vs. prefix match — if the server checks that the redirect starts with the registered value instead of matching it exactly, https://app.target.com.attacker.com or https://app.target.com/../evil might pass
  • Path traversal after a valid host — https://app.target.com/callback/../../oauth-relay if any open-redirect endpoint exists elsewhere on the same host
  • Subdomain wildcards registered too broadly — if *.target.com is accepted and an attacker controls or can find an XSS/open-redirect on any subdomain, that's enough
  • Missing scheme enforcement — some implementations forget to check that the scheme matches too, letting a registered https callback also accept an equivalent-looking non-https redirect in edge cases
GET /oauth/authorize?client_id=abc123
  &redirect_uri=https://app.target.com.attacker.com/callback
  &response_type=code&scope=profile HTTP/1.1
Host: auth.target.com

2. The state parameter, or the lack of one

state exists specifically to prevent CSRF against the OAuth flow itself. Without it — or with it present but never actually validated server-side — an attacker can start their own OAuth flow, capture the resulting authorization code, and trick a victim into completing it in their browser. The victim ends up with the attacker's identity linked to their account, or vice versa, depending on the flow. It's a quiet bug because the login still "works" from the victim's point of view; nothing looks broken.

Test it directly: start a flow, strip or reuse an old state value, and see if the callback still completes successfully.

3. Implicit flow token leakage

Implicit flow returns the access token directly in the URL fragment. Fragments aren't sent to the server in the Referer header by spec, but that protection disappears the moment any script on the page reads location.href and forwards it somewhere — analytics scripts are a common accidental culprit. If a client still uses implicit flow, check what third-party scripts run on the redirect page and whether any of them capture the full URL.

4. PKCE, and what happens without it

PKCE (Proof Key for Code Exchange) exists to stop an intercepted authorization code from being exchanged by anyone other than the client that requested it — originally for public clients like mobile and SPA apps that can't hold a secret safely. Where PKCE is missing or optional rather than enforced, an intercepted or leaked authorization code (via a referrer, a logging system, or a malicious app on the same device) can potentially be redeemed by someone other than the legitimate client. If the authorization server accepts a token exchange with a code but no matching code_verifier at all, that's worth flagging directly.

5. Scope escalation

Some authorization servers trust the scope parameter more than they should. Try requesting scopes broader than what the client normally asks for:

GET /oauth/authorize?client_id=abc123
  &redirect_uri=https://app.target.com/callback
  &response_type=code&scope=profile+email+admin:write HTTP/1.1

If the consent screen doesn't clearly show the requested scopes, or the server grants scopes the client was never approved for, that's a real finding — especially if it lets a low-trust integration obtain high-trust access.

6. Client secret exposure

Mobile apps and SPAs sometimes ship a "confidential client" secret directly in the compiled app or JS bundle because migrating to a public-client flow properly (PKCE, no secret) was more work. Decompiling an APK or reading a JS bundle for a hardcoded client_secret is a five-minute check that occasionally pays off big, since that secret combined with a loose redirect_uri check can sometimes let an attacker impersonate the legitimate client entirely.

Key takeaways
  • Test redirect_uri validation with subdomains, path traversal, and prefix-match tricks
  • Confirm state is both present and actually validated, not just decorative
  • Check for PKCE enforcement, especially on public clients
  • Grep mobile app binaries and JS bundles for hardcoded client secrets
