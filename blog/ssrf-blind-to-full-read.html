SSRF Hunting: From Blind Callbacks to Full Internal Read

Any feature that makes an outbound request on your behalf is a feature worth testing. Here's where those features hide and how I escalate them.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Find every feature that fetches a URL you supply, confirm the request fires with an out-of-band callback, then work through internal IPs, cloud metadata endpoints, and alternate schemes to see how far the app will let that request travel.

Server-Side Request Forgery is what happens when an application will fetch a URL on your behalf and you get to choose — even partially — what that URL points to. It's one of my favorite bug classes because the impact ranges from "mildly annoying" to "full read access to your internal network" depending entirely on how much the backend trusts its own outbound requests.

Where SSRF actually lives

Forget looking for a text box labeled "URL." The real candidates are usually features that don't look like networking at all:

  • Webhooks — "notify this URL when an event happens" is SSRF-by-design unless it's locked down
  • URL preview / link unfurling — chat apps and messaging features that show a thumbnail for a pasted link
  • PDF and document generators — anything built on a headless browser (wkhtmltopdf, Puppeteer) that renders attacker-supplied HTML or fetches remote assets
  • Image processing — "import avatar from URL" style features
  • SSO / SAML metadata URLs — some identity provider configs let you point at an arbitrary metadata endpoint
  • Import/export integrations — "connect your RSS feed," "import from this API," anything with a base URL field

Step 1: Prove the request fires at all

Before worrying about internal IPs, confirm the server actually makes an outbound request using a domain you control — either your own listener or an out-of-band interaction service. Point the vulnerable parameter at it:

POST /api/v1/webhooks HTTP/1.1
Host: app.target.com
Content-Type: application/json

{"callback_url": "http://your-oob-domain.example/probe-1"}

A hit on your listener confirms blind SSRF. No hit doesn't necessarily mean it's safe — check whether the request happens asynchronously (queued job, background worker) before giving up; those can take seconds to minutes to fire.

Step 2: See how far the leash goes

Once outbound requests are confirmed, the real question is what the backend can reach that a normal user can't. Things worth trying, scoped to what the target program explicitly allows:

  • Loopback and internal ranges — 127.0.0.1, 169.254.0.0/16, RFC1918 space, and the app's own internal service ports
  • Cloud metadata endpoints — 169.254.169.254 is the well-known metadata address on most major cloud providers, and depending on the platform and IMDS version in use, it can expose instance role credentials to anything running on the box
  • Alternate URL schemes — some HTTP client libraries happily follow file://, gopher://, or dict:// if the app doesn't explicitly restrict schemes
  • DNS rebinding — a hostname that resolves to a public IP at validation time and an internal IP at request time, useful against apps that validate the URL once and fetch it later
Scope this one carefully Poking at cloud metadata and internal services can have real consequences depending on what's actually running there. Stick to what the program's scope explicitly permits, stop as soon as you've proven impact, and never pull or exfiltrate more data than a screenshot or short snippet requires.

Bypassing naive filters

A lot of apps do attempt SSRF protection, just not very well. Common gaps:

  • Blocklisting localhost and 127.0.0.1 but forgetting 127.0.0.2, 0.0.0.0, or the IPv6 loopback ::1
  • Checking the hostname before a redirect, then blindly following a 3xx response to wherever it points
  • Validating the initial DNS resolution but not pinning the connection to that resolved IP (the DNS rebinding case above)
  • Decimal, octal, or hex-encoded IP representations that parse differently across libraries — e.g. 2130706433 as a decimal form of 127.0.0.1

Chaining for bigger impact

Blind SSRF that just proves an outbound request fired is a real but modest finding. It gets significantly more severe when you can chain it into:

  • Reading the response body back (via an error message, a rendered PDF, or an "import successful" preview) — turning blind SSRF into full internal read
  • Hitting an internal admin API with no auth because it "trusts requests from inside the network"
  • Retrieving temporary cloud credentials from a metadata endpoint and using them against the provider's own API, if in scope
Key takeaways
  • Any "fetch a URL for me" feature is worth testing, even ones that don't look like networking
  • Prove the request fires out-of-band before worrying about escalation
  • Check whether the app validates the URL once and fetches later — that's where DNS rebinding lives
  • Escalate toward reading the response back; blind is good, readable is better
