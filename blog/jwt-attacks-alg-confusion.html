JWT Attacks: alg Confusion, kid Injection, and Weak Secrets

These bugs were first written about years ago. I still find them in production today.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Before touching any of the classic JWT attacks, just read the header and claims. Half the time the interesting bug is sitting in plain sight — a weak secret, a permissive kid, or an alg the server shouldn't be honoring.

A JWT is three base64url-encoded segments — header, payload, signature — separated by dots. None of it is encrypted, all of it is trivially decodable, and the entire security model rests on the server correctly verifying that signature before trusting anything in the payload. Most JWT bugs come from libraries or custom code that get that verification step subtly wrong.

Start by just decoding it

echo "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9" | base64 -d
# {"alg":"HS256","typ":"JWT"}

Check the header for the algorithm in use and whether a kid (key ID) field is present, and check the payload for the claims being trusted — role, user ID, permissions, expiry. This alone tells you which of the attacks below are even worth attempting.

1. alg: none

The JWT spec technically allows an alg of none, meaning no signature at all. Some libraries, if not explicitly configured to reject it, will accept a token with {"alg":"none"} and an empty signature segment as valid:

header:  {"alg":"none","typ":"JWT"}
payload: {"sub":"1234","role":"admin"}
token:   eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiIxMjM0Iiwicm9sZSI6ImFkbWluIn0.

Note the trailing dot with nothing after it — no signature. If the server accepts this, you can set any claim you want, including a role of your choosing.

2. RS256 → HS256 algorithm confusion

When a server uses RS256, it verifies signatures with a public key and signs with a private key the client never sees. Some verification libraries, though, will happily verify an HS256 token too — and HS256 uses one shared secret for both signing and verifying. If an attacker can get the RS256 public key (often published at a /.well-known/jwks.json endpoint or embedded in a mobile app), they can craft their own token, sign it with HS256 using the public key as the HMAC secret, and if the server's verification code doesn't strictly pin the expected algorithm, it may treat that signature as valid.

The fix is on the server side — explicitly pin the expected algorithm during verification rather than trusting whatever the token's header claims. Worth testing whenever RS256 is in use and a public key is reachable.

3. kid header injection

The kid claim tells the server which key to use for verification, which is convenient for key rotation and a liability when the value is used unsafely:

  • Path traversal — if kid is used to build a file path (/keys/{kid}.pem), a value like ../../../../dev/null might point verification at a predictable or empty file, which combined with algorithm confusion can mean an attacker-known "key"
  • SQL injection — if kid is used in a database lookup without parameterization, classic SQLi payloads are worth testing there specifically
  • Arbitrary key via jku/x5u — some implementations honor a jku (JWK Set URL) or x5u (X.509 URL) header, fetching the verification key from a URL supplied inside the token itself. If the server doesn't strictly allow-list acceptable hosts, pointing it at a server you control means you supply both the token and the "trusted" key used to verify it.

4. Weak or guessable HMAC secrets

HS256 tokens are only as strong as the shared secret. Short, dictionary-word, or default secrets (frameworks that ship a placeholder value in example config that never gets changed) can be brute-forced offline once you have a single valid token:

jwt_tool eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjM0In0.abc123 \
  -C -d /usr/share/wordlists/jwt-secrets.txt

If the secret cracks, you can forge arbitrary tokens with any claims you like, signed correctly, indistinguishable from a legitimate one.

5. Missing expiry or signature verification entirely

Occasionally the simplest bug is true: the server decodes the JWT and reads claims from it without ever calling the actual verify function, or it verifies the signature but never checks exp. A token captured months ago might still work indefinitely. Always check both independently — a library call named verify doesn't guarantee every check you'd assume is bundled into it.

A note on scope Brute-forcing secrets and forging tokens directly affects authentication — treat it with the same care as any other account takeover testing, stick to your own test accounts, and stop as soon as impact is demonstrated.
Key takeaways
  • Decode the token first — the header alone often tells you what to try
  • Test alg confusion whenever RS256 is used and the public key is reachable
  • Treat kid as untrusted input; look for path traversal, injection, and jku/x5u abuse
  • Attempt an offline secret crack against HS256 tokens with a known wordlist
  • Verify that expiry is actually checked, not just the signature
