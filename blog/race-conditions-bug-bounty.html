Race Conditions in Web Apps: Winning the Bug Bounty Lottery

If a limit is checked and then applied in two separate steps, there's a window between them. Race condition bugs live in that window.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Any "check, then act" logic — check balance, then deduct; check redemption count, then redeem — has a race window. Fire the requests as close to simultaneously as possible and see if the check can be satisfied more than once.

Race conditions used to be a pain to exploit reliably over HTTP because network jitter meant your parallel requests rarely landed at exactly the same moment on the server. That got a lot easier once the "single-packet attack" technique became common knowledge — and it's now one of the highest reward-to-effort bug classes in bounty hunting, because so much backend logic still assumes requests arrive one at a time.

The pattern to look for

Almost every race condition bug follows the same shape in the backend code, something like:

1. SELECT balance FROM accounts WHERE id = ?
2. if balance >= amount:
3.     UPDATE accounts SET balance = balance - amount WHERE id = ?
4.     process_withdrawal(amount)

Between step 2 and step 3, the balance hasn't been touched yet. If you fire two withdrawal requests at the exact same instant, both can pass the check at step 2 before either one completes step 3 — and you get two withdrawals approved against a balance that should only have covered one.

Where this shows up in real features

  • Coupon and gift card redemption — redeeming a single-use code twice
  • Referral and signup bonuses — claiming a one-time reward multiple times
  • Rate limits and quotas — exceeding an "N actions per day" limit by submitting them in parallel
  • Account actions with uniqueness constraints — adding the same email as a team member twice before the first request's uniqueness check commits
  • Voting, liking, or review submission — bypassing "one action per user" logic

The single-packet technique

Sending parallel requests from a script and hoping they land close enough together is unreliable — connection setup and TLS handshakes introduce enough jitter that the race window often closes before your second request arrives. The more reliable approach, popularized by PortSwigger's research and implemented in Burp Suite's Turbo Intruder, is to open the underlying TCP connections ahead of time and hold the requests back until the very last byte, then release them within microseconds of each other over HTTP/2 or pipelined HTTP/1.1. That collapses network jitter almost entirely and makes the race window a matter of server-side processing time instead of network timing.

# Turbo Intruder — conceptual sketch of a race test
def queueRequests(target, wordlists):
    engine = RequestEngine(endpoint=target.endpoint,
                            concurrentConnections=1,
                            engine=Engine.BURP2)

    for i in range(20):
        engine.queue(target.req, gate='race1')

    engine.openGate('race1')  # release all queued requests together

Confirming the bug is real, not just noisy

Before reporting, I make sure the result is consistent and clearly beyond normal behavior:

  • Repeat the test a few times — a single lucky double-redemption could be a fluke; a reliable one at, say, 15 successful duplicate redemptions out of 20 attempts is a real bug
  • Confirm the end state is actually wrong (balance, redemption count, or database record), not just that two responses came back 200
  • Note whether the app uses a queue or job runner for the sensitive step — that changes where the race window actually is

Why this is still so common

Fixing race conditions properly means atomic database operations (a single UPDATE ... WHERE balance >= amount statement instead of separate read-then-write), row-level locking, or idempotency keys on the client request. All three require deliberate design decisions that "make the happy path work" development doesn't naturally produce, which is exactly why this bug class hasn't gone anywhere even as awareness of it has grown.

Key takeaways
  • Look for any "check a limit, then perform an action" logic
  • Use connection warming / single-packet techniques instead of naive parallel requests
  • Confirm the end state actually changed, not just that you got multiple 200 responses
  • Idempotency keys and atomic queries are the fix — call that out in your report
