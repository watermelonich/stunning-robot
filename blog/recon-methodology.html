My Recon Methodology for Bug Bounty Targets

Bugs get found by whoever looks at the right asset first. Good recon is how you make sure that's you.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Enumerate subdomains from as many sources as possible, probe them all for live HTTP services, screenshot and triage by what's actually running, then mine JavaScript for endpoints and parameters no one else bothered to read.

On a mature program, the login page has been tested by a thousand other researchers before you got there. The forgotten staging subdomain from three product pivots ago has been tested by maybe nobody. Recon is how you find the second thing instead of the first.

1. Subdomain enumeration, from every angle

No single source finds everything, so I run several in parallel and merge the results:

# passive sources
subfinder -d target.com -all -o subs_subfinder.txt
amass enum -passive -d target.com -o subs_amass.txt

# certificate transparency
curl -s "https://crt.sh/?q=%25.target.com&output=json" | jq -r '.[].name_value' | sort -u > subs_crtsh.txt

# merge + dedupe
cat subs_*.txt | sort -u > all_subs.txt

I also check Wayback Machine and gau for historical hostnames that might not resolve to anything obvious today but still point at a live service:

gau --subs target.com | unfurl domains | sort -u >> all_subs.txt

2. Resolve and probe for what's actually alive

A list of hostnames is just a list until you know which ones respond. httpx handles resolution, live-host probing, and useful metadata (status code, title, tech stack, response size) in one pass:

httpx -l all_subs.txt -sc -title -tech-detect -o live_hosts.txt

This turns thousands of raw hostnames into a few hundred confirmed live web services, each already tagged with what it's likely running.

3. Screenshot and triage

Reading a few hundred status codes and titles is doable. Scanning a wall of screenshots is faster, and it's how login portals, admin panels, and "under construction" pages jump out immediately:

gowitness file -f live_hosts.txt --write-db

I sort by what looks interesting first: anything that says "admin," "staging," "internal," or "dev" in the title or URL, anything running a visibly outdated tech stack, and anything that looks like it was stood up for a single feature and forgotten.

4. Read the JavaScript — all of it

Modern frontends ship most of the application's routing and API surface as JavaScript. That means the JS bundle is documentation the developers didn't mean to publish:

  • LinkFinder or a custom regex pass to pull out endpoint paths and full URLs from bundled JS
  • Grepping for API base URLs, especially ones pointing at internal-sounding hostnames like internal-api or admin-svc
  • Secret scanning across the same bundles — hardcoded keys, tokens, or internal service URLs show up more often than they should
for js in $(cat js_files.txt); do
  curl -s "$js" | grep -oE "(\"|')(\/api\/[a-zA-Z0-9_\-\/\.]+)(\"|')"
done | sort -u > discovered_endpoints.txt

5. Parameter discovery

Endpoints without known parameters aren't fully mapped yet. arjun and historical URL mining fill that gap:

arjun -u https://app.target.com/api/search -oT params_found.txt
gau target.com | grep "?" | unfurl keys | sort -u >> params_found.txt

Hidden or debug parameters (?debug=1, ?admin=true, ?format=json) surface surprisingly often this way, and they're worth testing even when the "normal" version of the endpoint looks solid.

6. Turn it into a repeatable pipeline

None of this is useful if you run it once. New subdomains and endpoints appear constantly, so the last step is stitching the above into something that runs on a schedule and only shows you what's new since the last run — a simple diff against the previous output is enough to turn recon into a standing advantage instead of a one-time sweep.

Respect scope Passive sources and light active probing (DNS resolution, HTTP status checks) are generally low-risk, but always check the program's scope and rules of engagement before running anything heavier, and never scan assets explicitly listed as out of scope.
Key takeaways
  • Merge multiple subdomain sources — passive, certificate transparency, and historical archives all find different things
  • Screenshots make triage across hundreds of hosts fast
  • JS bundles are an endpoint map developers didn't intend to publish — read them
  • Automate the whole pipeline and diff against previous runs to catch new attack surface early
