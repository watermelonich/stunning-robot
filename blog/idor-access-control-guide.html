Breaking Access Control: A Practical Guide to Finding IDORs

Insecure Direct Object References are unglamorous, unfashionable, and still one of the most reliable ways to find a critical in any program with an ID in the URL.

Heads up: Educational content only. Only test systems you're explicitly authorized to test, and always follow the rules of the program you're hacking on.
TL;DR Create two low-privilege test accounts, map every object reference in the app (not just numbers in URLs), and try to read, write, and delete Account B's data while authenticated as Account A. Do it for every endpoint, not just the obvious ones.

IDOR — Insecure Direct Object Reference — is the bug class I've reported more than any other, and it's not close. It doesn't need a fancy payload, it doesn't need a WAF bypass, and it doesn't care how modern your frontend framework is. It just needs a backend that checks "is this user logged in?" without ever asking "is this user allowed to touch this specific object?"

Automated scanners are bad at finding these because the request looks completely valid. The status code is 200. The response is well-formed JSON. Nothing about the traffic screams "vulnerability" unless you know whose data you're supposed to be looking at.

Step 1: Build a proper test matrix

Before touching a single endpoint, I set up:

  • Two accounts at the lowest available privilege tier (Account A, Account B)
  • One account at a higher tier if the app has roles (admin, org owner, support agent)
  • Objects created under each account — orders, documents, messages, invites, API keys, whatever the app deals in

The goal is to have two authenticated sessions where I know exactly which objects belong to which account. Without that, you can't tell an authorization bug from a bug in your own test setup.

Step 2: Map every object reference, not just the URL

Numeric IDs in the URL path are the easy case. The bugs that pay better hide in:

  • Request bodies — {"invoice_id": 4821} buried in a POST
  • Query strings — ?export=user_4821.csv
  • Headers — some apps pass a resource ID in a custom header for "performance reasons"
  • GraphQL node IDs — often base64-encoded, e.g. SW52b2ljZTo0ODIx decodes to Invoice:4821
  • WebSocket messages — easy to miss if you're only watching the HTTP tab
  • File names and exports — PDFs, CSVs, and ZIPs that get generated with a predictable or sequential name

Step 3: The swap test

With Account A authenticated, request an object you know belongs to Account B. Here's the generic shape of the test:

GET /api/v1/invoices/48213 HTTP/1.1
Host: app.target.com
Cookie: session=<account-A-session>

HTTP/1.1 200 OK
Content-Type: application/json

{"id":48213,"owner":"account-A","total":"1,204.00","pdf_url":"/files/invoice_48213.pdf"}

Now increment the ID to one that belongs to Account B and repeat the exact same request, same Account A session:

GET /api/v1/invoices/48214 HTTP/1.1
Host: app.target.com
Cookie: session=<account-A-session>

HTTP/1.1 200 OK
Content-Type: application/json

{"id":48214,"owner":"account-B","total":"389.50","pdf_url":"/files/invoice_48214.pdf"}

If that second response comes back 200 with Account B's data, you've got a horizontal access control bug. If Account A is a regular user and you can pull data that should require an admin role, that's vertical privilege escalation, and it usually scores higher.

Don't stop at read

Read-only IDOR gets triaged. Read-write IDOR gets paid. Always follow up the GET test with:

  • PUT / PATCH — can you modify Account B's object?
  • DELETE — can you remove it entirely?
  • State-changing actions — can you cancel Account B's order, revoke their API key, or add yourself to their team?
Scope carefully Write and delete tests against another account's real data are exactly the kind of thing that can cause real damage. Use disposable test accounts and objects you created yourself whenever a program allows it, and never run destructive tests against a target you don't fully control unless the program scope explicitly permits it.

Where IDOR actually hides

After a while you start noticing patterns in where this bug class clusters:

  • Support and chat transcripts — ticket IDs are rarely checked as carefully as the main app
  • Notification and activity feeds — /api/notifications/{id}/mark-read is a classic
  • Bulk/export endpoints — CSV and PDF generation code is often bolted on later and skips the auth middleware everything else uses
  • Admin panels exposed to regular staff roles — internal tools built fast, with authorization as an afterthought
  • Anything behind a "share" or "public link" feature — predictable tokens are IDOR wearing a disguise

Tooling that speeds this up

Manual testing is how you learn the app, but at scale I lean on:

  • Burp Suite's Autorize / Auth Analyzer extensions — replay every request with a second session's cookie automatically
  • A small Python match-and-replace script against a saved Burp history export, useful when the object reference is buried somewhere Autorize doesn't reach
  • diffing responses between the "should fail" and "known good" request to catch subtle authorization checks that return 200 with an empty or redacted body instead of a proper 403/404

Writing it up

The report that gets paid fastest includes: the exact request that should have been rejected, the two account identities involved, a screenshot or short clip showing Account A viewing Account B's data, and a one-line explanation of impact ("any authenticated user can read every other user's invoices by incrementing this ID"). No filler, no exploit theory — just proof.

Key takeaways
  • Test every object reference, not just numeric IDs in the URL
  • Always test write and delete, not just read
  • Look in exports, notifications, and support tooling first — that's where checks get forgotten
  • A clean two-account proof beats a long paragraph every time
